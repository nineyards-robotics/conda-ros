# conda-ros

Pre-built ROS2 conda packages, inspired by [RoboStack](https://robostack.github.io/).

This repo contains the build pipeline, recipes, patches, and snapshot pin files for distributing ROS2 packages as conda binaries. It supports multiple ROS2 distributions from a single repo.

## How it works

### Source of truth: rosdistro

All package versions come from the [ros/rosdistro](https://github.com/ros/rosdistro) repository. Each ROS2 distribution (Humble, Jazzy, Rolling, etc.) has a `distribution.yaml` that tracks released versions of every package. Each entry points to a **release repo** (typically under `ros2-gbp/`) at a specific version, containing source code and packaging metadata generated by [bloom](https://github.com/ros-infrastructure/bloom).

Release repos produce per-distro releases — the source for `nav2=1.2.0` on Jazzy may differ from `nav2=1.2.0` on Humble, since most packages maintain separate branches per distro. This means recipes and patches are inherently per-distro, per-version.

### Snapshots

A **snapshot** is a coherent set of packages built together from a rosdistro release tag (e.g. `jazzy/2026-01-28`). These tags represent package sets that have passed upstream integration testing — a stronger compatibility guarantee than arbitrary time-based sampling.

The output of a snapshot build is:
- **Built conda packages** published to a channel
- **A pin file** listing the exact `name=version=buildstring` for every package

The rosdistro tag is an input to recipe generation, not persistent state — the generated packages and pin file *are* the snapshot.

### Pin files vs lockfiles

A snapshot's **pin file** is not a lockfile — it's a set of constraints. It lists every ROS package at its exact `version=buildstring` but does not include non-ROS dependencies or represent a solved environment.

Downstream tooling uses the pin file to constrain ROS packages to their snapshot versions, then solves the full environment including non-ROS dependencies (pytorch, numpy, etc.) against both the ROS channel and conda-forge. The buildstring pin is essential — it's what guarantees ABI consistency, since it encodes which dependency versions a package was compiled against. Pinning by version alone is not sufficient.

### Snapshot lifecycle

Snapshots are maintained on `main`. If a package has a build failure, we fix it (add a patch, update the recipe), rebuild, and update the pin file in place. Git history tracks what changed.

Old snapshot pin files remain available so teams can stay locked to a known-good set. Built packages persist in the channel indefinitely. Teams roll forward by changing the snapshot they reference.

Old snapshots are not patched retroactively. If a team needs a newer package version against an older snapshot, that's handled by rebuilding the package against the old snapshot's dependency set.

### Build pipeline

The build pipeline is implemented as Python modules in `pipeline/`, invoked via pixi tasks.

#### Step 1: Fetch snapshot metadata

Fetch rosdistro at the release tag and parse `distribution.yaml` to get every package's release URL, version, and tag format.

#### Step 2: Generate snapshot build config

Start with the root `conda_build_config.yaml` (system pins), add every ROS package version from `distribution.yaml`, and write the merged result to `distros/{distro}/snapshots/{date}/conda_build_config.yaml`. This is the build config used for all packages in this snapshot — dependency versions in this file feed into the build hash via rattler-build's variant system.

#### Step 3: Generate recipes

For each package in `distribution.yaml`:
- If `packages/{pkg}/{version}/recipe.yaml` already exists, skip — the version hasn't changed since a previous snapshot
- If new version: fetch release repo source, parse `package.xml`, generate recipe
- Handles build type detection (ament_cmake, ament_python, cmake), rosdep key → conda mapping, vendor package passthrough, and implicit dependency injection

#### Step 4: Resolve build order

Build a dependency graph from all recipes and topological sort to get the build order. This ensures that when a package is built, all of its dependencies are already available in the channel.

#### Step 5: Build in dependency order

For each package in topological order:
- Use rattler-build with `--skip-existing=all` against the remote channel
- The snapshot's `conda_build_config.yaml` provides variant resolution — dependency versions feed into the hash
- rattler-build computes the hash, checks the channel, and skips if the artifact already exists
- If not skipped: apply any patches from `packages/{pkg}/{version}/patches/`, build, and push to channel
- Track failures and continue — best-effort, build the largest working subset

This is where incremental builds happen naturally. If `rclcpp` bumped between snapshots, every package that depends on it gets a new hash (because the rclcpp version in `conda_build_config.yaml` changed), misses the cache, and rebuilds. Packages whose dependencies didn't change produce the same hash and are skipped.

#### Step 6: Publish and record

- Push built artifacts to the remote public channel
- Record every successfully built package's `name=version=buildstring` in `distros/{distro}/snapshots/{date}/pins.yaml`

#### Step 7: Verify

Create a temporary pixi project with all pins as constraints and solve against the channel + conda-forge. This proves the full package set is installable as a coherent environment — catching any conflicts between ROS packages or with system dependencies that individual builds wouldn't surface.

#### Step 8: Commit

- Commit recipes, snapshot build config, and pin file to `main`

### Patches

Patches live at `packages/{pkg}/{version}/patches/` and apply to a specific version of a package's source code. They are applied during the build step.

Patches evolve over time on `main` — if a dependency change in a new snapshot causes a build failure for an existing package version, the fix is a new patch committed to that version's directory. This doesn't affect old snapshots because their artifacts are already built and in the channel. Old snapshots are never rebuilt; only the current snapshot is actively maintained.

If the package bumps to a new version in a future snapshot where the issue is fixed upstream, the patch is simply irrelevant — it stays with the old version directory and the new version gets its own clean recipe.

**Important:** patches and recipe changes don't affect the build hash — the hash is computed purely from variant config (dependency versions). To force a rebuild after adding a patch or modifying a recipe, bump the build number in `recipe.yaml`:

```yaml
build:
  number: 1  # was 0
```

The build number is part of the buildstring (`hABCDEF_0` → `hABCDEF_1`), so `--skip-existing` will see it as a new artifact and rebuild. Convention: reset to 0 on new package version, increment by 1 for recipe/patch changes at the same version.

### Package versioning

Every conda package is identified by three components:

```
ros-jazzy-nav2 = 1.2.0 = hABCDEF_0
                  ^         ^
                version   build string
```

- **Version** — upstream package version from `distribution.yaml`
- **Build string** — hash derived from variant configuration values used by the recipe (dependency versions pinned via `conda_build_config.yaml`) plus a build number. Only variant keys that the recipe actually references feed into its hash.
- **Build number** — incremented for packaging-level fixes that don't change the version or dependency hash

The build hash is content-addressed: if nothing about a package or its dependencies changed, the same artifact is reused. If a dependency bumps, downstream packages get new build hashes and are rebuilt automatically.

## Repo structure

```
conda-ros/
├── pixi.toml                   # Environment and task definitions
├── rosdep.yaml                 # rosdep key → conda package mapping
├── conda_build_config.yaml     # System dependency version pins
├── pipeline/                   # Build pipeline (Python, invoked via pixi tasks)
│
└── distros/
    └── {distro}/               # e.g. jazzy, humble, rolling
        ├── distro.yaml         # Distro config: package selection, overrides
        ├── packages/
        │   └── {package}/
        │       └── {version}/
        │           ├── recipe.yaml
        │           └── patches/
        │               └── fix-something.patch
        └── snapshots/
            └── {YYYY-MM-DD}/
                ├── pins.yaml
                └── conda_build_config.yaml
```

### Why per-distro, per-version?

Packages are organized by distro, then by package name and version, because:

- Release repos produce **per-distro releases** — the source for a given package version can differ across distros
- Patches apply to specific source code and are therefore version-specific
- Recipes encode version-specific metadata (source URL, tag, dependencies)
- Old versions must remain available for the custom version pinning service to rebuild against newer snapshots

This mirrors how rosdistro release repos work — old release tags stick around alongside new ones — but as browsable directories rather than git branches.

### Shared config

Some configuration is shared across distros:

- **`rosdep.yaml`** — maps rosdep keys to conda package names. Most mappings are universal. Platform-specific overrides use `linux`/`osx`/`win` sub-keys.
- **`conda_build_config.yaml`** (root) — pins system dependency versions (compilers, protobuf, spdlog, etc.) shared across snapshots. Tracks conda-forge's `conda-forge-pinning` feedstock to maximize compatibility with the broader conda ecosystem.

Each snapshot also gets its own **`conda_build_config.yaml`** under `snapshots/{date}/`, which merges the root system pins with all ROS package versions for that snapshot (generated from `distribution.yaml`). This is what makes the build hash work — rattler-build only hashes variant config entries that a recipe actually uses, so each package's hash reflects exactly the dependency versions it was built against, with no unnecessary coupling to unrelated packages. Keeping the per-snapshot build config means the custom pinning service has everything it needs to rebuild old packages.

## Platforms

Target platforms:

- `linux-64`
- `linux-aarch64` (Jetson, Raspberry Pi, ARM SBCs)
- `osx-64`
- `osx-arm64` (Apple Silicon)
- `win-64`

Recipes are mostly platform-agnostic, using selectors for platform-specific differences. Platform count does not multiply the directory structure.

## Commit conventions

Commits use structured tags in the message body so changes are machine-searchable. A script in `pipeline/` generates these tags by diffing the working tree.

### Tag format

```
[action][scope...]
```

Where scope narrows from distro → package → version as applicable:

| Action | Scope | Example |
|--------|-------|---------|
| `recipe-add` | distro, package, version | `[recipe-add][jazzy][nav2][1.3.0]` |
| `recipe-update` | distro, package, version | `[recipe-update][jazzy][nav2][1.2.0]` |
| `patch-add` | distro, package, version | `[patch-add][jazzy][nav2][1.2.0]` |
| `patch-update` | distro, package, version | `[patch-update][jazzy][nav2][1.2.0]` |
| `patch-remove` | distro, package, version | `[patch-remove][jazzy][nav2][1.2.0]` |
| `snapshot-add` | distro, date | `[snapshot-add][jazzy][2026-03-15]` |
| `snapshot-update` | distro, date | `[snapshot-update][jazzy][2026-03-15]` |
| `rosdep-update` | — | `[rosdep-update]` |
| `build-config-update` | — | `[build-config-update]` |
| `distro-config-update` | distro | `[distro-config-update][jazzy]` |
| `pipeline-update` | — | `[pipeline-update]` |

### Example commits

A snapshot build:
```
jazzy/2026-03-15: add snapshot

[snapshot-add][jazzy][2026-03-15]
[recipe-add][jazzy][nav2][1.3.0]
[recipe-add][jazzy][rclcpp][28.1.4]
```

A build fix:
```
fix nav2 1.2.0 build on jazzy

[patch-add][jazzy][nav2][1.2.0]
[snapshot-update][jazzy][2026-03-15]
```

## Relationship to RoboStack

This project is inspired by and builds on [RoboStack](https://robostack.github.io/). Key similarities:

- ROS packages built with conda-forge dependencies
- rosdep key mapping for dependency resolution
- rattler-build for package building

Key differences:

- Snapshots derived from rosdistro release tags (vs RoboStack's mutex-based epoch system)
- Incremental builds using content-addressed build hashes (vs full-distro rebuilds)
- Per-version recipe and patch storage enabling custom version pinning
- Pipeline kept in-repo, invoked via pixi tasks (vs separate vinca tool)
